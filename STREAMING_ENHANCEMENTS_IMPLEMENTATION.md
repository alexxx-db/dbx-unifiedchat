3. **`reasoning_complete`** - When reasoning is done
   ```python
   writer({
       "type": "genie_progress",
       "agent": agent.space_id,
       "status": "reasoning_complete"
   })
   ```

4. **`completed`** - When tool finishes successfully
   ```python
   writer({
       "type": "genie_progress",
       "agent": agent.space_id,
       "status": "completed",
       "result_size": len(answer)
   })
   ```

5. **`failed`** - When tool encounters an error
   ```python
   writer({
       "type": "genie_progress",
       "agent": agent.space_id,

   })
   ```

## 🔧 Technical Implementation Details

### Synchronous Pattern (Correct Approach)

Following official Databricks/MLflow documentation:
- ✅ Uses synchronous `Generator[ResponsesAgentStreamEvent, None, None]`
- ✅ Uses `app.stream()` not `app.astream()`
- ✅ Tools use regular `def` not `async def`
- ✅ No async/await complexity

**Why Synchronous?**
- Databricks Model Serving handles concurrency at infrastructure layer
- All official MLflow examples use sync patterns
- Simpler, more maintainable code
- Better compatibility with existing systems

### Error Handling

Graceful fallback pattern:
```python
try:
    writer = get_stream_writer()
except Exception:
    writer = None

if writer:
    writer({...})  # Only emit if writer available
```

This ensures tools work correctly even if:
- Not running in LangGraph context
- `get_stream_writer()` is unavailable
- Streaming is disabled

### Config Propagation

Tools accept `config: RunnableConfig = None` parameter:
- Passes streaming context to sub-agents
- Enables nested tool streaming
- Required for `get_stream_writer()` access
